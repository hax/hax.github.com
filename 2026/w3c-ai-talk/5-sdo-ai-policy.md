## 标准组织的 AI 政策

```
标准组织
也开始
定政策
```

再来，我们看看标准组织。

```
TC39
2024-02
issue #62
```

TC39 这条线可以从 `tc39/code-of-conduct` 仓库的 issue `#62` 讲起。这个讨论是 2024 年 2 月发起的，标题就是要不要明确禁止 generated content。

```
TC39
2026-03-20
AI_POLICY.md
```

到 2026 年 3 月 20 日，`tc39/how-we-work` 仓库里加入了 `AI_POLICY.md`。
这份 policy 要求参与者在 TC39 仓库里发的 issues 和 comments 不能用 AI 生成。

```
只管 prose
不管 code
```

它同时明确写了: 这条 policy 不适用于代码贡献；代码还是按原来的 review process 处理。对 prose 的要求是，内容必须是你自己写的，不能把 LLM 扩写后的文字直接贴进讨论区。机器翻译和 proofreading 可以用，但不能引入新内容。

```
must be
your own writing
```

`AI_POLICY.md` 的原文就是这么写的: `Prose contributions and comments must be your own writing`。

```
ISO
2025-03
AI guidance
```

ISO 这边对应的是 2025 年 3 月的 `Guidance on use of artificial intelligence (AI) for ISO committees`，版本号是 `Version 1.1 March 2025`。

```
ISO
研究
翻译
会议记录
```

这份 guidance 允许把 AI 用在研究、翻译、会议记录这些场景。

```
GenAI 图文
别进 ISO 内容
```

但同一份文档也写得很直接: 不要把生成式 AI 生成的图片或文字放进 ISO 内容；也不要把 ISO 受版权保护的材料喂给免费的、公共的、或者没有授权的 AI 工具。

```
Do not use
images or text
created by generative AI
in any ISO content
```

这也是 ISO guidance 里的原文句子。它针对的不是讨论区，而是 ISO 内容本身。

```
Linux Foundation
legal guidance
```

Linux Foundation 这边对应的是法律团队发布的 `Guidance Regarding Use of Generative AI Tools for Open Source Software Development`。

```
review
no differently
```

这份 guidance 里有一句很核心的话: `development and review ... should be treated no differently`。也就是说，它没有要求把 AI 生成代码单独放进另一套审查流程。

```
但要确认
条款与归因
```

它补充的要求是另外两条: 先确认工具条款不会和项目的开源许可或 IP 政策冲突；如果输出里包含第三方版权材料，要补上相应的许可、notice 和 attribution。

```
W3C
2026-02-03
Disclosure CG
```

W3C 这条线不是从禁令开始，而是从一个新的 Community Group 开始。`AI Content Disclosure Community Group` 的 call for participation 发布于 2026 年 2 月 3 日。

```
人写
AI 辅助
AI 写人审
全自动
```

这个社区组想讨论的是，内容使用生成式 AI 的程度，能不能用结构化、机器可读的方式表达出来。它举的就是这类分级。

```
制度层
不只是
工具层
```

如果把这些组织放在一起看，虽然关心的内容和政策范畴不同，但有一点启示：AI 带来的变化，最后一定会进入制度层，而不只是停留在工具层。
